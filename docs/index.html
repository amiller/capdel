The wager

A protocol is real when independent implementations agree.

One implementation can be self-consistent and still wrong — it silently defines the parts the spec left vague. A second, independent implementation has no access to the first one's private assumptions. Where the two diverge is exactly where the specification was thin. Where they agree without negotiation, the spec is load-bearing.

So capdel now ships as two brokers — a stdlib-only Python reference and a dependency-free Deno / TypeScript port — plus a language-neutral conformance suite that drives both over the same wire protocol and asserts identical verdicts. Same vectors in, same allow / deny out.

What has to agree

The sharp edge is the cryptography.

Allow/deny logic is forgiving — two readings of a rule usually still reject the same attack. Cryptography is not: the two brokers must build the exact same bytes before hashing, or a token signed by one is garbage to the other. This canonicalization ambiguity is the classic under-specification bug — the entire pain history of JWT / JOSE. Pinning it is the whole point of a second implementation.

The interop proof

One broker mints it, the other verifies it — on shared state.

The strongest check isn't running two brokers in parallel; it's making them hand a live token to each other. A proof-of-possession capability minted by Python, sitting as a file on disk, is verified and attenuated by Deno — then the capability Deno derived is verified back by Python.

  1. python
    mints a PoP capability p — its token is an HMAC key, written to a file on disk.
  2. deno
    reads the same state dir and verifies a token-signed request against p — canonical string and on-disk format agree.
  3. deno
    attenuates p into a narrower child capability and returns its derived key.
  4. python
    independently recomputes that child key from the formula — identical, byte-for-byte.
  5. python
    verifies the child capability Deno just minted. Full round trip, both directions.

The line it draws

Authority is specified. Enforcement is a platform concern.

The suite makes an architectural boundary explicit. What two implementations can agree on — and what belongs in the wire spec — is authority semantics. What can't be cross-validated, because it's platform-specific by nature, is enforcement. Keeping them apart is the design, not a shortcut.

in the spec · conformance-tested

Authority

Minting, attenuation subset rules, invocation allow/deny, proof-of-possession, escalation, revocation, trusted closure. Portable across languages — and now proven so.

platform annex · out of scope here

Enforcement

Deno permissions, kernel Landlock / seccomp, sandboxing. What actually stops an unconfined agent from ignoring the broker. Specific to where it runs — a separate, honest problem.

capdel is an early prototype. This report covers the authority layer: the two brokers agree on it completely. The enforcement layer above is deliberately not claimed here. The conformance suite lives in the repo and is the regression gate for every future change to the spec.
